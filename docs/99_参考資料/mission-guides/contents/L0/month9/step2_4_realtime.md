## 会議前の準備（最重要）

### やること一覧

| タイミング | やること |
|-----------|---------|
| **会議の前日** | アジェンダ（議題）を確認する |
| **会議の30分前** | テンプレートを開き、基本情報を記入する |
| **会議の直前** | 前回の議事録を確認し、持ち越し事項を把握する |

### 具体的な準備

```markdown
# 開発定例ミーティング 議事録

## 基本情報

| 項目 | 内容 |
|------|------|
| 日時 | 2025-04-08 10:00 - 11:00 |        ← 事前に記入
| 場所 | Zoom |                              ← 事前に記入
| 参加者 | （会議開始時に記入） |
| 記録者 | 山田太郎 |                         ← 事前に記入

## アジェンダ

1. 前回アクションアイテムの確認               ← 事前に記入
2. API仕様のレビュー                         ← 事前に記入
3. リリース準備の進捗確認                     ← 事前に記入

## 前回からの持ち越し事項                      ← 前回議事録から転記
- テスト環境の構成（鈴木が調査結果を報告予定）
```

> 事前準備で基本情報とアジェンダが埋まっていれば、会議中は議論内容に集中できます。

## 会議中のメモ術

### 原則: 完璧を目指さない

会議中は **キーワードと要点だけ** をメモします。文章にするのは会議後でOKです。

### メモの取り方

```markdown
## 議論内容

### 1. 前回アクションアイテムの確認

- 田中: テスト完了 OK
- 佐藤: ライブラリ選定 → Auth0に決定
- 鈴木: ステージング環境 → 遅延、4/10に延期

### 2. API仕様のレビュー

- エンドポイント一覧を画面共有
- /api/users GET → OK
- /api/users POST → バリデーションルール追加必要
  - メールの形式チェック
  - パスワード8文字以上
- 佐藤: 仕様書を修正して再共有する

### 3. リリース準備

- 4/15リリース予定 → 変更なし
- チェックリスト確認
  - DB マイグレーション → 田中担当
  - 本番環境設定 → 鈴木担当
```

### メモのコツ

| コツ | 説明 |
|------|------|
| **発言者を書く** | 「田中:」のように名前を先頭に |
| **矢印を使う** | `→` で結果や結論を示す |
| **略語でOK** | 後で分かれば省略してよい |
| **決定事項に印** | `[決定]` や `★` で目印をつける |
| **分からないことはメモ** | `[?]` を付けて後で確認 |

---

## 会議中に使える速記テクニック

```
★ または [決定]  → 決定事項
→              → 結果、結論
[TODO]          → アクションアイテム
[?]             → 後で確認が必要
[持越]          → 次回に持ち越し
cf.             → 参考情報
```

### 速記の実例

```markdown
### API仕様レビュー
- POST /api/users のバリデーション [?] 佐藤に確認
- ★ メール形式チェックを追加する [決定]
- [TODO] 佐藤: 仕様書修正 (4/10)
- エラーレスポンスの形式 [持越] → 次回議論
```

## 会議後の清書

会議が終わったら、**できるだけ早く**（理想は30分以内に）メモを清書します。

### 清書のステップ

```
1. メモを読み返す
   ↓
2. 決定事項を整理する（★や[決定]を探す）
   ↓
3. アクションアイテムをテーブルにまとめる（[TODO]を探す）
   ↓
4. 未決定事項を整理する（[持越]や[?]を探す）
   ↓
5. 議論内容を箇条書きで整理する
   ↓
6. チェックリストで確認する
   ↓
7. 参加者に共有する
```

### 清書の例

会議中メモ:
```markdown
- POST バリデーション ★メール形式チェック追加
- [TODO] 佐藤 仕様書修正 4/10
- エラーレスポンス [持越]
```

清書後:
```markdown
## 決定事項
- POST /api/users にメール形式チェックのバリデーションを追加する

## アクションアイテム

| 担当 | 内容 | 期限 |
|------|------|------|
| 佐藤 | API仕様書のバリデーション部分を修正 | 4/10（木） |

## 未決定事項

| 議題 | 次のアクション |
|------|---------------|
| エラーレスポンスの形式 | 次回MTGで議論 |
```

## 共有のタイミングと方法

| 共有方法 | タイミング | 適切な場面 |
|----------|-----------|-----------|
| Slackで投稿 | 会議後30分以内 | 日常的な定例会議 |
| GitHubにpush | 会議後1時間以内 | 開発に関する会議 |
| メールで送信 | 会議後半日以内 | 社外参加者がいる会議 |

> 早く共有するほど、記憶が新しいうちにフィードバックがもらえます。
