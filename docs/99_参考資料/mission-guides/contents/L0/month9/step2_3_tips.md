## コツ1: 決定事項を明確に書く

### 悪い例

```markdown
## 決定事項
- 認証方式について話し合った
- リリース日を検討した
```

問題点: 「話し合った」「検討した」では、何が決まったか分かりません。

### 良い例

```markdown
## 決定事項
- 認証方式は **OAuth2.0** を採用する
- リリース日は **4月15日（火）** とする
```

> ポイント: 「何を」「どうする」を具体的に書く。重要な部分は太字にする。

---

## コツ2: アクションアイテムには「担当者」と「期限」を必ず付ける

### 悪い例

```markdown
## アクションアイテム
- テストを実施する
- ドキュメントを更新する
- 環境を準備する
```

問題点: 誰がいつまでにやるのか分かりません。

### 良い例

```markdown
## アクションアイテム

| 担当 | 内容 | 期限 |
|------|------|------|
| 田中 | 結合テストの実施 | 4/7（月） |
| 佐藤 | API仕様書の更新 | 4/4（金） |
| 鈴木 | ステージング環境の構築 | 4/7（月） |
```

> ポイント: テーブル形式で「担当」「内容」「期限」の3点セットを必ず書く。

---

## コツ3: 要約は箇条書きで

### 悪い例

```markdown
## 議論内容
田中さんがログイン画面の実装が完了したと報告しました。現在テスト中で、
来週の金曜日までにはテストも終わる見込みです。佐藤さんはAPI設計を
進めていて、仕様に関する質問があるとのことでした。鈴木さんは
インフラの構築を進めていて、80%ほど完了しているそうです。
```

問題点: 長い文章は読みにくく、ポイントが埋もれます。

### 良い例

```markdown
## 議論内容

### 進捗報告
- **田中**: ログイン画面の実装完了。テスト中（4/11完了予定）
- **佐藤**: API設計を進行中。仕様に関する質問あり → 別途MTGで解決
- **鈴木**: インフラ構築 80%完了。残りはステージング環境
```

> ポイント: 箇条書きにして、名前を太字にすると一目で誰の話か分かる。

## コツ4: 「事実」と「意見」を区別する

### 悪い例

```markdown
- OAuth2.0が良いと思う
```

問題点: 誰の意見か分からず、決定なのか意見なのか曖昧。

### 良い例

```markdown
- 佐藤: セキュリティの観点からOAuth2.0を推奨
- 田中: 独自認証はメンテナンスコストが高いため、OAuth2.0に賛成
- **結論: OAuth2.0を採用**（全員合意）
```

> ポイント: 発言者を明記し、最終的な結論を太字で目立たせる。

---

## コツ5: 未決定事項も記録する

決まらなかったことも重要な情報です。

```markdown
## 未決定事項（持ち越し）

| 議題 | 理由 | 次のアクション |
|------|------|---------------|
| テスト環境の構成 | 情報が不足 | 鈴木が調査して次回報告 |
| UI のデザイン | デザイナー不在 | 4/3のMTGで確認 |
```

> ポイント: 「なぜ決まらなかったか」「次にどうするか」をセットで書く。

---

## コツ6: 略語や専門用語は初出時に補足する

会議に参加していない人が読むことを前提に書きましょう。

### 悪い例

```markdown
- SSOをIdPと連携してSAMLで実装する
```

### 良い例

```markdown
- SSO（シングルサインオン）をIdP（認証プロバイダー）と連携し、
  SAML（認証プロトコル）で実装する
```

> ポイント: 全員が同じ知識レベルとは限りません。初出の略語は説明を添えましょう。

## まとめ

| コツ | 内容 |
|------|------|
| 決定事項を明確に | 「何を」「どうする」を具体的に |
| アクションに3点セット | 担当者 + 内容 + 期限 |
| 箇条書きで要約 | 長文を避け、ポイントを箇条書きに |
| 事実と意見を区別 | 発言者を明記、結論を太字で |
| 未決定も記録 | 持ち越し事項は次のアクション付きで |
| 略語に補足 | 初出時は説明を添える |
